# Skill: Brainstorming

Use at the start of any non-trivial task to explore the problem space before committing to a solution.

## When to use

Any task where the approach is not immediately obvious, where there are trade-offs to surface, or where the request is ambiguous.

## Steps

1. **Restate the goal** in your own words. If it doesn't match the request, ask for clarification before continuing.
2. **Identify constraints** — what must not change, what is already decided, what has been ruled out.
3. **List 2–4 viable approaches** with explicit trade-offs for each. Cut anything you won't seriously consider.
4. **State a recommendation** with the primary reason. Do not present options without a recommendation.
5. **List open questions** that only the user can answer before you can proceed.

## Output format

- One sentence restating the goal.
- Constraints (bullet list).
- Approaches with trade-offs (short table or bullet pairs).
- Recommendation + reason.
- Open questions (if any).

## What not to do

- Do not start implementing during brainstorm. This step ends with a direction, not code.
- Do not list more than 4 approaches.
- Do not skip the recommendation. "Here are 3 options" is not helpful — pick one.
