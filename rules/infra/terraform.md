# Terraform Conventions

Rules for Terraform modules, resources, and state management. The `infra-engineer` agent applies these to every Terraform output.

---

## Module structure

```
module/
├── main.tf          # resource definitions
├── variables.tf     # input variable declarations
├── outputs.tf       # output value declarations
├── versions.tf      # required_version + required_providers with version constraints
└── README.md        # inputs/outputs table (generated by terraform-docs)
```

- One `.tf` file per logical concern — do not put everything in `main.tf`.
- For large modules, split by resource group: `networking.tf`, `compute.tf`, `iam.tf`.
- Keep modules small and focused. A module that does everything is hard to reuse and test.

---

## Versioning

- Pin Terraform itself and every provider:
  ```hcl
  terraform {
    required_version = ">= 1.8, < 2.0"
    required_providers {
      aws = {
        source  = "hashicorp/aws"
        version = "~> 5.50"
      }
    }
  }
  ```
- Never use `version = "*"` or omit the version constraint.
- Update providers deliberately, not automatically. Review changelogs for breaking changes.

---

## State management

- Always use a remote backend in production. Never leave state in local `terraform.tfstate`:
  ```hcl
  terraform {
    backend "s3" {
      bucket         = "my-org-tf-state"
      key            = "prod/main.tfstate"
      region         = "us-east-1"
      dynamodb_table = "tf-state-lock"
      encrypt        = true
    }
  }
  ```
- Enable state file encryption and versioning on the S3 bucket.
- Use DynamoDB for state locking — prevents concurrent `apply` operations from corrupting state.
- Never commit `.tfstate` or `.tfstate.backup` files to git.

## Variables and outputs

- Declare a `description` and `type` for every input variable:
  ```hcl
  variable "instance_type" {
    description = "EC2 instance type for the API service."
    type        = string
    default     = "t3.micro"
  }
  ```
- Mark sensitive variables `sensitive = true` — this suppresses them from plan output:
  ```hcl
  variable "db_password" {
    description = "RDS master password."
    type        = string
    sensitive   = true
  }
  ```
- Never hardcode sensitive values (passwords, tokens, keys) in `.tf` files. Inject via CI environment variables or a secrets manager.
- Add `sensitive = true` to outputs that expose secret values.

## Naming and tagging

- Use consistent, predictable resource naming: `<org>-<env>-<service>-<resource>` (e.g., `myorg-prod-api-sg`).
- Tag every resource that supports tags. Minimum required tags:
  ```hcl
  tags = {
    Environment = var.environment
    Service     = var.service_name
    ManagedBy   = "terraform"
    Owner       = var.team
  }
  ```
- Use a `locals {}` block for shared tag maps to avoid repetition.

---

## IAM (security-critical)

- Apply least privilege to every IAM policy. No `"Action": "*"` or `"Resource": "*"` without explicit justification signed off by `security-reviewer`.
- Use `data "aws_iam_policy_document" {}` instead of inline JSON strings — it is type-checked by Terraform and easier to review.
- Separate IAM definitions into `iam.tf`. Do not scatter IAM inline in compute or storage resources.
- Flag any IAM changes for a `security-reviewer` pass before applying.

---

## Secrets

- Do not store secret values in Terraform variables passed as plain text — they end up in state.
- Prefer referencing secrets from AWS Secrets Manager or SSM Parameter Store at runtime:
  ```hcl
  data "aws_secretsmanager_secret_version" "db_password" {
    secret_id = "prod/api/db-password"
  }
  ```
- Never output a raw secret value. If an output must reference a secret, mark it `sensitive = true`.

## Plan and apply discipline

- Always run `terraform plan` before `terraform apply`. Review the plan diff carefully.
- In CI/CD: run `plan` on PR open, run `apply` only on merge to the protected branch.
- Use `terraform plan -out=tfplan` to save the plan and `terraform apply tfplan` to apply exactly what was reviewed.
- `terraform destroy` is intercepted by the Aegis security hook and triggers a user confirmation dialog — only a human can approve it. Use `-target=<resource>` for surgical teardown.
- Enable `prevent_destroy = true` on resources that must not be accidentally deleted (databases, S3 buckets with data, production VPCs).

---

## Code quality

- Run `terraform fmt -recursive` before every commit. CI must fail on unformatted files.
- Run `terraform validate` in CI to catch syntax and type errors.
- Run `tflint` to catch provider-specific warnings that `validate` misses.
- Run `checkov` or `tfsec` for security policy checks before merge.
- Use `terraform-docs` to regenerate `README.md` from variable/output blocks on every change.

---

## Workspaces

- Use workspaces only for lightweight environment separation where state differences are small (e.g., dev vs. staging with the same config).
- For significantly different environments (different accounts, different regions, different resource counts), use separate state files in separate directories or modules — not workspaces.
- Never run `terraform workspace select prod` by hand in a dev session. Production applies must go through CI only.
